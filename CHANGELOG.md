# Changelog

## 2.3.x ŌĆö 2026-05-02

- Change: Removed the trailing `ŌĆ”` placeholder rendered next to folded headings (`.heading-with-toggle.is-folded::after`). The chevron (Ō¢Č when folded, Ō¢╝ when expanded) already signals fold state, so the ellipsis was redundant visual noise. CSS-only; no round-trip impact.
- Change: TOC sidebar now starts collapsed on the first open and remembers the user's last toggle across files and reopens. Previously `useState(false)` always forced the panel open on every editor open, which ate screen width before the user had a chance to interact. The new `betterMarkdown.tocCollapsed` globalState key (same pattern as `betterMarkdown.cursors` / `headingFolds`) is hydrated through the existing `init` message and written back on every click / drag-threshold crossing. First-run (no value stored) defaults to collapsed, matching the explicit feature request.
- Bug fix: YouTube / GitHub embed Enter/Escape no longer leaves the caret stranded inside a freshly-created embed. `YouTubeEmbedView.exit()` / `GitHubEmbedView.exit()` first called `save()` (which dispatches `updateAttributes({ url })` ŌĆö async, doesn't synchronously mutate the node) and then guarded cursor placement on `node.attrs.url`, which was still the stale empty value at that point. The guard kept short-circuiting on first commit, so the caret was never moved out of the embed. `save()` now returns the trimmed URL it just committed, and `exit()` guards on that local value instead of the stale node snapshot. Applies to both `webview/extensions/YouTubeEmbed.tsx` and `webview/extensions/GitHubEmbed.tsx`.
- Bug fix: Rich editor now picks up external `.md` changes (git pull, peer editors, format-on-save by other extensions) even when the change lands while the webview is hidden. Previously the host posted `update` only from `onDidChangeTextDocument`; if the webview was hidden when that fired, VS Code could drop the message and the editor stayed stuck on the pre-change content after switching back. The custom editor's `onDidChangeViewState` now re-syncs the document on becoming visible: it diffs against `lastSentContent` (a snapshot of the last payload the webview is known to have received) and posts an `update` whenever the content drifted. The docChange handler intentionally does not advance the snapshot while the webview is hidden, so the resync still detects the drop and recovers. The `pendingWebviewEdits` guard is honored so the webview never receives its own in-flight edit back.
- Bug fix: Heading fold chevron and `ŌĆ”` placeholder now flip correctly on unfold. Toggling a fold dispatched a meta-only ProseMirror transaction (`docChanged === false`), which Tiptap's `ReactNodeViewRenderer.update()` short-circuits ŌĆö so the `HeadingView` React component never re-rendered after the plugin state changed. The chevron stayed `Ō¢Č` and the CSS-generated `ŌĆ”` ellipsis (from `.heading-with-toggle.is-folded::after`) persisted even though the hidden children were already revealed. `HeadingView` now subscribes to the editor's `transaction` event and force-updates whenever a `HEADING_FOLD_KEY` meta is observed, so the chevron and the `is-folded` class flip in the same tick as the fold/unfold.
- Feature: Cmd+Opt+K from the rich editor now inserts the current file as an `@`-mention into Claude Code (file-level only ŌĆö line ranges remain a follow-up). Claude Code's `claude-code.insertAtMentioned` reads `vscode.window.activeTextEditor` directly, which is `undefined` while a custom editor is focused, and its own keybinding is gated on `editorTextFocus` (false in the webview). The new `betterMarkdown.claudeCodeInsertAtMentioned` command ŌĆö bound to `Cmd+Opt+K` with `when: activeCustomEditorId == betterMarkdown.editor` ŌĆö resolves the active custom-editor tab's URI, calls `showTextDocument` with an empty selection to make the source `TextEditor` active (so the upstream command emits `@<relpath>` rather than `@<relpath>#Lstart-Lend`), then executes `claude-code.insertAtMentioned`. Restricted to `file:` URIs (git:/scm: read-only views are a no-op) and wrapped in try/catch so the keystroke degrades gracefully when Claude Code isn't installed.
- Feature: Heading fold/unfold toggle. Every heading now shows a small chevron in the left gutter (Ō¢╝ when expanded, Ō¢Č when folded) ŌĆö click it to collapse everything between this heading and the next same-or-higher-level heading, Notion / outline style. Implemented as a Tiptap `Heading` NodeView for the toggle plus a ProseMirror plugin that produces `display: none` `Decoration`s for the hidden range, so the underlying document ŌĆö and the markdown round-trip ŌĆö is untouched. Fold state is persisted per file via VS Code's `globalState` (same mechanism as `betterMarkdown.cursors`), so reopening a file restores whichever headings were collapsed. State is keyed by heading *index*, which means inserting or removing a heading above a folded one may shift the fold to the wrong row until it's re-toggled ŌĆö a knowingly-accepted MVP trade-off.
- Bug fix: Clicking a `.md` file in the Source Control "Changes" view (and any other diff-editor entry point ŌĆö the diff-editor toolbar, the inline "vs HEAD" path) now opens VS Code's native text diff editor instead of the Markdown Studio rich editor. Cause: our `customEditors` contribution has priority `"default"` for `*.md`, which is the right choice for normal file opens but also caused both panes of a markdown diff to render in the rich editor webview, hiding the actual diff. The previous mitigation in `onDidChangeTabs` (auto-closing non-`file:` custom-editor tabs) papered over one side of the leak but never surfaced the diff. We now also detect `TabInputTextDiff` tabs whose original or modified URI is a `.md` and reopen them via `vscode.diff(..., { override: "default" })`, which forces the editor resolver to bypass custom editors. A single-use `Set` keyed on the URI pair prevents the reopen from triggering itself recursively. Users who prefer the explicit escape hatch can still set `"workbench.editorAssociations": { "{git,vscode-scm}:/**/*.md": "default" }` ŌĆö but it's no longer required.
- Security: Removed the `Open in Browser` command and the bundled localhost HTTP/WebSocket server (`server/index.ts`, `dist/server.js`). Several attack vectors lived in that server ŌĆö shell injection via the WS `openLink`/`toggleEditor` handlers (`open "${href}"` / `code "${path}"`), path traversal in the `/doc/<base64dir>/<file>` route, arbitrary writes via `/upload/<base64dir>/<filename>`, and unauthenticated WebSocket access to `saveSettings`. The feature also required `ws` (and `@types/ws`) as direct dependencies and exposed `localhost:3333` on every machine that had ever invoked the command. Removing the command, its CodeLens entry, the webview "Open in Browser" button, and the WebSocket browser-mode shim eliminates the attack surface in one stroke; the rich editor stays a custom-editor experience inside VS Code. Tracked in `docs/security-audit/2026-06-02-audit.md` (C-1 through C-4, M-5, M-6, H-4).
- Bug fix: Ctrl+A no longer steals focus into a math node. Tiptap's React node-view marks an atom as `selected` whenever the current selection range covers it (`from <= pos && to >= pos + nodeSize`), and `AllSelection` from select-all does exactly that ŌĆö so every math node flipped to `selected=true`, which a `useEffect` was using to auto-enter editing mode and focus the input/textarea. The first math node would win the focus race and Ctrl+A would land inside it instead of selecting the document. Auto-edit now only fires for an exact `NodeSelection` targeting the node (i.e. arrow-key nav into the atom); clicks still go through the explicit `onClick` on the rendered KaTeX. Applied to both `MathInline` and `MathBlock`.
- Bug fix: Bubble menu no longer re-renders on every ProseMirror transaction, fixing a per-keystroke perf regression introduced in 2.3.4. The menu's effect listened to `transaction` and unconditionally called `forceUpdate`, so the 488-line component reconciled on every keystroke even when no button state changed. Now the listener computes a small signature (selection range + every `isActive(...)` value the menu reads ŌĆö bold/italic/strike/code/link, paragraph/blockquote/lists/codeblock, and headings 1ŌĆō6) and only re-renders when the signature flips. Visibility is unchanged: showing/hiding is owned by Tiptap's `BubbleMenu` via `shouldShow`, which is unaffected. Net result: noticeably snappier typing on long documents.
- Feature: Inline math now triggers from typing `$...$` in the editor. Closing the second `$` (with non-space at both ends, e.g. `$E=mc^2$`) collapses the run into an inline math node ŌĆö no more reaching for the slash menu. Currency phrasings like `$5 to $10` are left alone (a non-space-at-both-ends guard) and `$$...$$` block math doesn't get half-consumed (negative lookbehind on the opening `$`). Implemented as a Tiptap `InputRule` on the `MathInline` extension.
- Feature: Settings now live in VS Code's native config under the `markdownStudio.*` namespace and appear in the Settings UI (search "Markdown Studio"). All 16 settings can be edited there, synced via Settings Sync, or overridden per-workspace via `.vscode/settings.json`. The in-app ŌÜÖ panel still works and writes to User scope. A one-time migration on first activation copies pre-existing globalState settings into User-scope config so nobody loses their configuration; the legacy `betterMarkdown.settings` globalState entry is cleared after the copy. The namespace is `markdownStudio.*` (not `betterMarkdown.*`) because VS Code's Settings UI auto-derives section labels by title-casing dotted segments ŌĆö `betterMarkdown.bullet` would have rendered as "Better Markdown: Bullet" regardless of `configuration.title`. Internal command IDs and the `betterMarkdown.editor` view type are unchanged. Closes #4.
- Refactor: Extracted the bubble menu's URL editor into its own component (`webview/components/BubbleLinkEditor.tsx`). State, callbacks, and JSX that previously lived inline inside `EditorBubbleMenu.tsx` now belong to the focused child; the parent only keeps a `linkMode` boolean and an `onClose` handler. No behavior change.
- Feature: Restored the Link button in the bubble menu ŌĆö this time as an inline URL editor instead of a `window.prompt`. Click the Link button (or press it via keyboard nav) and the bubble's button row swaps for a URL input flanked by Cancel / Apply / Remove. The input is auto-focused and pre-selected, so paste-and-Enter applies the link in one motion. Pressing Escape returns to the formatting row without applying. If the selection already has a link, the existing href is prefilled and the Remove button appears. Restored `BTN_LINK` (index 4) and bumped `BTN_TURN_INTO`/`BTN_COUNT` back to 5/6 so keyboard arrow-counting matches.
- Bug fix: Bubble menu no longer jumps a few pixels every time you click Bold / Italic / Strike. Toggling a mark reflows glyph metrics, so `view.coordsAtPos(to)` returned a slightly different x for the same selection range on each call, and the menu re-anchored on every transaction. Now the anchor is cached by `(from, to)` and stored relative to the editor DOM ŌĆö bold/italic clicks reuse the cached offset (menu stays put) while scroll and resize still update because we re-read the editor's bounding rect on every read.
- Change: Bubble menu now opens below the end of the selection (anchored to a zero-size virtual rect at the trailing caret position with `placement: "bottom-start"`) instead of floating above the highlight. The menu used to sit on top of the selected text and obscure it; now the highlighted run stays fully visible while you click formatting buttons.

## 2.1.x ŌĆö 2026-04-20

- Bug fix: Math and GFM checkboxes in the rich diff view now show before/after with the standard red strike + green underline, instead of just the new version. Cause: htmldiff's token-key function only captures attributes for `<img src>`, `<a href>`, `<object data>`, `<iframe src>`, and the trio `<svg>/<math>/<video>` (compared as the full token); every other tag collapses to just `<tagname>`, so `<span data-latex="a^2">` vs `<span data-latex="b^2">`, and `<input type="checkbox">` vs `<input type="checkbox" checked>`, both look identical to it. We now wrap each math node and each checkbox in a `<video>` sentinel before handing the HTML to htmldiff (chose `<video>` over the more obvious `<math>`/`<svg>` because it's the only one of the three that DOMParser parses as a regular HTML element ŌĆö no foreign-content namespace switch). htmldiff sees changed sentinels as different tokens and emits a `<del>`/`<ins>` pair around the wrapper; a post-processing pass replaces each sentinel with the actual rendered KaTeX `<span>`/`<div>` or the actual `<input type="checkbox">`. Side effect: the older `math-diff-changed` blue tint added in 2.1.28 is gone ŌĆö the standard ins/del coloring now applies to math just like the rest of the document.
- Bug fix: Rich diff view now KaTeX-renders math (`$...$` and `$$...$$`) instead of showing the raw LaTeX source. The markdown-to-HTML pipeline used by the diff (`markdownToDisplayHtml`) emits `<span data-type="mathInline" data-latex="...">` / `<div data-type="mathBlock" ...>` placeholders meant to be picked up by Tiptap's NodeView, but the diff view stuffs the HTML straight into the DOM with `dangerouslySetInnerHTML` and never ran KaTeX. KaTeX rendering now happens inside the post-processing pass that unwraps the `<video>` sentinels (see above).
- Bug fix: Diff view no longer shows a duplicated, undiffed copy of the document trailing the diff. Cause: `.diff-view` was `position: absolute; inset: 0` inside `.editor-container` (which is `overflow-y: auto`), so it only covered the visible viewport. As soon as the user scrolled past the diff's height, the underlying Tiptap editor ŌĆö still mounted as a sibling of the DiffView ŌĆö became visible, looking like the entire document had been re-pasted at the bottom without diff marks. Switched the overlay to `position: fixed; inset: 0` so it pins to the webview viewport regardless of editor-container scroll state.
- Bug fix: Diff toolbar no longer slides up by half its height the first time Prev/Next is pressed. Cause: `el.scrollIntoView({ block: "center" })` walks the entire ancestor chain and scrolls every overflow:auto container that needs to scroll ŌĆö including `.editor-container` outside the diff ŌĆö which dragged the toolbar partway off-screen. Replaced with a manual `body.scrollTo({ top })` computed from `getBoundingClientRect`, scoped to `.diff-body` only. Added `flex-shrink: 0` to `.diff-toolbar` defensively so it can never compress under flex pressure.
- Bug fix: GFM checkboxes inside the rendered diff now show the Notion-style custom box even when `htmldiff` has wrapped the `<input>` in `<ins>`/`<del>`. The CSS used a direct-child combinator (`li.task-list-item > input`) which broke as soon as a parent `<ins>` was introduced; switched to a descendant selector.
- Bug fix: Inline math whose content starts with a digit (e.g. `$24 \times 5 = 120$`) now renders as math instead of literal text. Cause: `protectCurrencyDollars` matched any `$<digit>` and replaced the leading `$` with a placeholder so unrelated currency amounts (`$14B ŌĆ” $1.4B`) wouldn't be paired by `remark-math`. That same rule consumed the opener of valid inline math, leaving the closing `$` orphaned and the whole span emitted as plain text. Replaced the blind regex with a per-line scan that, for each `$<digit>`, looks ahead for a closing `$` and only protects when the content between them doesn't look like math (no `\command`, no sub/superscript, no equation-form `=`, no operator-with-letter pattern, and no markdown bold markers).
- Change: Consolidated the two setup-related commands into one. `Markdown Studio: Open Setup Prompt` (which only re-showed an informational modal) and `Markdown Studio: Rerun First-Run Setup` (which only cleared a flag) were both narrow and overlapping. Replaced with `Markdown Studio: Factory Reset Settings` (`betterMarkdown.factoryReset`) ŌĆö a single destructive command that wipes the `betterMarkdown.settings` and `betterMarkdown.consentShown` globalState entries, syncs `settingsUpdated` to every open webview so they re-merge with defaults, and shows a modal confirmation before applying. The on-open welcome modal still fires automatically the first time a markdown file is opened post-install (no command needed).
- Change: Settings-panel toggles now support an optional secondary `description` line rendered under the label in muted text. Used on the `autoSave` toggle: the label is now a short `Save normalization on open (Recommended)` and the rationale (round-trip stability, why dirty-on-open is the alternative) lives on its own line beneath. The `Toggle` component takes a new `description?: string` prop and the layout switched to `align-items: flex-start` with a vertical text stack.
- Bug fix: `Open in Browser` command now works from inside the rich editor. Cause: `vscode.window.activeTextEditor` returns `undefined` when the focused editor is a custom editor (the rich Markdown view), so the URI lookup fell through to the "no markdown file" warning even when a `.md` file was clearly open. Fall back to the active tab's input URI before consulting `activeTextEditor`, mirroring the resolution `Toggle Rich/Source Editor` already uses.
- Change: First-run setup prompt is now an in-editor modal (`SetupPrompt.tsx`) rendered inside the rich editor instead of a VS Code information notification, so it can't be missed in the bottom-right corner. The host posts `showSetupPrompt` to the webview after `ready` (so the listener is attached) and the webview posts `setupPromptChoice` back; `applySetupChoice` on the provider just records consent. Dismissing via Esc / click-outside / ├Ś is treated as `Keep defaults`. The modal offers two actions ŌĆö `Review settings` (opens the ŌÜÖ panel) and `Keep defaults` ŌĆö the earlier `Disable all` button was removed because it conflated independent toggles and gave a misleading impression that the underlying md ŌåÆ html ŌåÆ md round-trip could be turned off; per-toggle control still lives in the ŌÜÖ panel.
- Feature: First-run consent prompt for normalization. - Bug fix: bold-around-code abutting a word (e.g. `**\`bold code\`**Apples**`, no space) no longer saves with `**A**`-style numeric character entities cluttering the source. remark-stringify emits those references because CommonMark flanking rules disallow from closing directly into a word when the run contains a code span ŌĆö without the entity, the bold disappears on re-open. Cause: Tiptap's default`Code`mark sets`excludes: '\_'`, which strips every other mark when code is applied ŌĆö so `ŌĆ”`was parsed as plain`ŌĆ”`on the way in. - Feature: Mermaid diagram support. ` ```mermaid ` fences render as live diagrams inline; edit the source and the preview updates. ## 1.0.0 ŌĆö 2026-04-08

### Rich Editing

- WYSIWYG markdown editor with full GFM support
- Headings (h1-h6), bold, italic, code, links, images, blockquotes, horizontal rules
- Tables with floating toolbar to add/delete rows and columns
- Task lists with Notion-style custom checkboxes
- Ordered, unordered, and nested lists with correct round-trip formatting
- Code blocks with syntax highlighting and language selector
- Math support ŌĆö inline (`$...$`) and block (`$$...$$`) with KaTeX rendering, slash commands, and click-to-edit LaTeX source
- Slash command menu ŌĆö type `/` at the start of a line to insert blocks (headings, lists, tables, code blocks, math, images)
- Copy selection as markdown source (Cmd+C / Cmd+X)
- Configurable serialization settings (bullet style, emphasis markers, HR character, normalizations)

### Navigation

- Table of contents sidebar with filtering, drag-to-resize, and truncation
- Sticky headings ŌĆö current section heading pins to the top while scrolling
- Find in document (Cmd+F / Ctrl+F) with regex and case-sensitive modes

### Diff

- Rich diff view ŌĆö compare working copy against HEAD, inline or standalone panel
- Rendered mode with word-level green/red highlights and prev/next hunk navigation (j/k)
- Source mode with line-level diff
- Available from editor toolbar, SCM context menu, diff-editor toolbar, and command palette

### Round-trip Fidelity

- h4-h6 preserved via trailing HTML comment metadata
- Table cell code-span pipes handled without double-escaping
- Unicode-aware underscore unescaping
