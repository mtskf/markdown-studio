## 0. エグゼクティブサマリー

**結論: WebView 本体は堅牢だが、`server/index.ts`（"Open in Browser" 機能の同梱ローカルサーバ）に致命的脆弱性が集中**している。最短かつ最大の対策は **`betterMarkdown.openInBrowser` コマンドとサーバ機能を丸ごと削除**すること。

| カテゴリ                     | 重大度      | 件数  | 主因                                                                                                      |
| ------------------------ | -------- | --- | ------------------------------------------------------------------------------------------------------- |
| コード実行・任意ファイル書込           | Critical | 4   | `server/index.ts`（shell injection × 3, パストラバーサル × 2, 認証なし listen）                                       |

## 1. Critical 所見（要即時対処）

### C-1. WebSocket `openLink` ハンドラの shell injection

- **場所**: `server/index.ts:365-376`
- **コード**: ``exec(`open "${href}"`)`` / Linux は `xdg-open`、Windows は `start`
- **攻撃**: WS クライアントが送る `href` を quote のみで連結。`href = "x\"; rm -rf ~; #"` で任意コマンド実行
- **シナリオ**: ユーザーが一度でも `Open in Browser` を実行すると `localhost:3333` で WS リスナーが立ち上がり、悪意あるブラウザタブ（CSWSH）か LAN 上の第三者が接続して任意コード実行
- **修正**: `spawn(opener, [href], { shell: false })` に置換

### C-2. 画像アップロードでの任意ディレクトリ書込

- **場所**: `server/index.ts:247-279` (`POST /upload/<base64dir>/<filename>`)
- **攻撃**: `dir` は base64url 経由で攻撃者制御可能、検証なし。`/etc/cron.d/`, `~/.bashrc`, `~/.ssh/authorized_keys` 等にサーバ権限で任意バイト列を書き込み可能
- **修正**: 起動時の `state.dirPath` 集合へのメンバーシップ検査

### C-3. `/doc/<base64dir>/<file>` でローカル任意ファイル取得

- **場所**: `server/index.ts:282-289`
- **攻撃**: base64url で絶対パスを与えて任意ファイル取得。LAN 経由でも到達可能
- **修正**: 許可ディレクトリ集合 + `path.resolve` 後の `startsWith` 検査

### C-4. 認証なし・全インターフェース listen + WS Origin 検証なし

- **場所**: `server/index.ts:419` (`server.listen(PORT)` の host 未指定)
- **攻撃**: Node.js のデフォルトで IPv4/IPv6 全インターフェースに bind 可能性あり。さらに WS は Origin 未検証で CSWSH 成立 → C-1 と組み合わせて RCE
- **修正**: `server.listen(PORT, "127.0.0.1")` + 起動時ランダムトークン必須 + WS の `req.headers.origin` 検証

> 注: C-4 のバインド挙動は調査エージェント間で見解差あり（Agent2 は「全インターフェース」、Agent4 は「Node デフォルトで `localhost` 解決」）。**実機で `lsof -i :3333` で確認推奨**。いずれにせよ明示的 loopback 化が安全。

## 2. High 所見

### H-1. リンク経由でワークスペース外の任意ファイルを VSCode で開ける

- **場所**: `src/provider.ts:256-270` (`openLink` メッセージ)
- **問題**: `path.resolve(docDir, href)` 後にスキーム検証や confine なしで `vscode.commands.executeCommand("vscode.open", targetUri)`。`../../../../etc/passwd` 形式の `href` で escape 可能
- **修正**: 許可リスト方式 (`http`/`https`/`mailto`) + `targetPath.startsWith(docDir + path.sep)` 検証

### H-2. 画像アップロード（拡張本体側）で `~/.bashrc` 上書き可能

- **場所**: `src/provider.ts:282-309` (`uploadImage` メッセージ)
- **問題**: `path.basename` で `..` は剥がれるが、**任意の拡張子・ファイル名で任意バイナリ書込が可能**。悪意ある `.md` を `~` で開かせ、postMessage 一発で `~/.bashrc` 等を上書き
- **修正**: ホワイトリスト拡張子 (`png|jpg|jpeg|gif|webp|svg`) + 内容ハッシュベースのファイル名

### H-3. `localResourceRoots` がワークスペース全体を許可

- **場所**: `src/provider.ts:181-186`
- **問題**: WebView から `vscode-resource://` 経由で workspace 内の任意ファイル（`.env` 等）を読める。`img-src https:` 許可と組み合わせて exfil 経路成立
- **修正**: `docFolder` のみに限定。複数フォルダ参照が必要ならドキュメント親ディレクトリのみ

### H-4. WS `saveSettings` で他クライアントの設定を無検証上書き

- **場所**: `server/index.ts:353-362`
- **問題**: 任意ペイロードを `~/.better-markdown-settings.json` に丸ごと書き、他クライアントへブロードキャスト。パース時例外 (DoS) や巨大ペイロードによるリソース消費が成立
- **修正**: `SETTING_KEYS` ホワイトリスト + 型検証

## 6. 推奨アクション（フォーク時の優先順）

### 第1優先: 攻撃面の根絶

1. **`betterMarkdown.openInBrowser` を完全削除**
   - `package.json:90` のコマンド定義
   - `src/extension.ts:130-193` のサーバ spawn ロジック
   - `src/provider.ts:271-275` のメッセージハンドラ
   - `server/` ディレクトリ
   - `esbuild.js:25-31` の `serverBuild`
   - `dist/server.js` 同梱解除
   - これだけで C-1〜C-4, M-5, M-6, H-4 が一掃される
1. **`capabilities.untrustedWorkspaces` を宣言**
   ```json
   "capabilities": {
     "untrustedWorkspaces": { "supported": "limited", "description": "..." },
     "virtualWorkspaces": { "supported": "limited", "description": "..." }
   }
   ```

### 第2優先: 拡張本体側の越権を塞ぐ

1. **H-1 修正**: `src/provider.ts:256-270` の `openLink` を許可リスト (`http`/`https`/`mailto`) に変更
2. **H-2 修正**: `src/provider.ts:282-309` の `uploadImage` に拡張子ホワイトリスト + ハッシュベースのファイル名
3. **H-3 修正**: `src/provider.ts:181-186` の `localResourceRoots` を `docFolder` のみに限定
4. **CSP 強化**: `img-src` から `https:` を削除（`src/provider.ts:385`, `src/diffPanel.ts:194`）。YouTube サムネが不要なら H-3 と合わせて完全ローカル化

### 第3優先: サプライチェーン削減

1. **`lucide-react` を inline SVG に置換**（~150 行、1 日工数）

### 第4優先: ハードニング

## 7. 監査の検証手順（再現性のため）

- **コード実行系**: `grep -nE 'eval\(|new Function|child_process|exec[A-Z]*\(|spawn|innerHTML|outerHTML|dangerouslySetInnerHTML|insertAdjacentHTML|v-html'` を `src/`, `webview/`, `server/` に対し実行
- **秘密情報**: `grep -rE '<patterns above>'` + `find` で秘密ファイル + `git log --all -p -S` で履歴 grep
- **外部通信**: `grep -nE 'fetch\(|http\.request|https\.request|axios|got|node-fetch|XMLHttpRequest|WebSocket|new Worker'` を全コードに対し実行
- **テレメトリ**: `grep -nE 'telemetry|posthog|mixpanel|segment|sentry|amplitude|google-analytics|plausible|matomo|TelemetryReporter|machineId|sessionId|installDate'`
- **サプライチェーン**: `package-lock.json` を直接パース、npm registry に `https://api.npmjs.org/downloads/point/last-week/<pkg>` と `https://registry.npmjs.org/<pkg>` で問い合わせ、GHSA は `https://api.github.com/advisories?ecosystem=npm&affects=<pkg>`
